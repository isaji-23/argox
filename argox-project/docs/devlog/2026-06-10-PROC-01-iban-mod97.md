# [PROC-01] Add ISO 13616 mod-97 validation to IBAN detection

- **Date:** 2026-06-10
- **PR:** #133  ·  **Branch:** fix/PROC-01-iban-mod97-validation
- **Status:** in-review

## What changed

- `_valid_iban` in `argox/processors/pii.py` now verifies the ISO 13616
  mod-97 checksum in addition to the existing shape check: the first four
  characters move to the end, letters map to 10–35, and the resulting
  integer must be congruent to 1 mod 97.
- Detector docstrings updated: IBAN joins CREDIT_CARD (Luhn), ES_DNI/ES_NIE
  (control letter) and IPV4 (octet range) as post-validated entities.
- Tests: an IBAN-shaped token failing mod-97 is no longer redacted; a valid
  non-Spanish (DE) IBAN still is.

## Why

Mod-97 was deliberately deferred in v1 on the assumption that shape alone
gave high precision. In practice any `XX99` + 11–30 alphanumerics token —
order references, document IDs — collided with the shape and got redacted.
The checksum removes those false positives at negligible cost and was the
only checksum-carrying entity still unvalidated. Came out of evaluating
PII-library adoption (Presidio): checksum validation closes most of the
precision gap without taking the dependency.

## Notes / follow-ups

- The same hardening (Luhn, mod-97, DNI control letter, E.164 phone) was
  applied to the Collector's residual PII scan in PR #132 (COL-07).
- Presidio remains the candidate if NER-grade detection (names, addresses)
  becomes a compliance requirement; the `Detector` protocol already allows
  plugging it in without changing the processor surface.
